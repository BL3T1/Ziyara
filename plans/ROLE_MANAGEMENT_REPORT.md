## 2. Role Attributes and States

Roles in the system are governed by a set of immutable and mutable states.

### 2.1 Technical Attributes
- **`is_system_role` (Boolean)**: If true, the role is pre-defined and cannot be deleted.
- **`is_deletable` (Calculated)**: Always the inverse of `is_system_role`.
- **`level` (Integer)**: Used for hierarchy checks (e.g., a Manager cannot modify a higher-level role).

### 2.2 Operational States
- **ACTIVE**: The role is functional and assigned to users.
- **PENDING_REASSIGNMENT**: A custom role that has been marked for deletion. It remains in a "read-only" state while its users are migrated to a new role.

## 3. Interaction Flows (Role Lifecycle)

The management of roles is restricted exclusively to the **Super Admin** via the `sys_` module (Core Security).

### 3.1 Creating a Custom Role
1.  **Interaction**: Super Admin navigates to "Role Management" and selects "New Custom Role".
2.  **Input**: Role Name, Description, and a selection from the global Permission Catalogue.
3.  **Process**: The system validates the permissions. Note: "Locked" permissions (e.g., `sys:db_write`) cannot be assigned to custom roles.
4.  **Result**: New role record created with `is_system_role = false`.

### 3.2 Modifying Permissions
- **System Roles**: Permissions for `SUPER_ADMIN`, `CEO`, etc., are protected. Certain core permissions cannot be removed to prevent system lockout.
- **Custom Roles**: Super Admin can replace the entire permission set at any time. Updates propagate instantly as JWTs are validated against the database or a short-lived cache.

### 3.3 The Deletion & Reassignment Workflow
To prevent "orphaned users", a custom role cannot be deleted if it is assigned to active accounts.
1.  **Trigger**: Super Admin clicks "Delete" on a custom role.
2.  **Check**: System detects `{N}` users currently assigned to this role.
3.  **Prompt**: Super Admin must select a "Target Role" for reassignment.
4.  **Execution (Atomic)**:
    - Update all users to the Target Role.
    - Create an Audit Log record.
    - Physically delete the custom role record.

## 4. Permission Matrix Logic

The system uses a `resource:action` pattern (e.g., `provider:approve`).

### Types of Permissions
1.  **Domain Permissions**: (e.g., `hotel:update`) Enforced within specific business modules.
2.  **Cross-Cutting Permissions**: (e.g., `sys:audit_read`) Enforced by the core system.
3.  **Restricted Data Permissions**: (e.g., `pay:view_commission`) Specifically restricted to G1 and G5 levels; hidden from G3 (Finance Manager) by default.
